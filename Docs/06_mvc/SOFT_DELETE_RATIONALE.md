# Обоснование выбора слоя для soft delete

Логика мягкого удаления реализована в **сервисном слое** (`PostService`), а не в контроллере.

## Почему сервис, а не контроллер

- Контроллер по заданию должен оставаться «тонким» и только делегировать вызовы — как в лекции.
- Правила «удалённый пост невидим для клиента» — это **бизнес-логика**, а не HTTP-обработка.
- Сервис централизует проверки: фильтрация в `all()`, проверка флага в `getById`, `save`, `removeById`.

## Почему не только репозиторий

- Репозиторий отвечает за **хранение** данных (`markRemoved`, `findById`, `save`).
- Решение «404 для удалённого поста» — правило предметной области, его место в сервисе.
- Так проще добавить отдельное admin-API для восстановления постов без изменения репозитория.

## Разделение ответственности

| Слой | Ответственность |
|------|-----------------|
| `PostController` | HTTP-маршрутизация, сериализация JSON |
| `PostService` | Бизнес-правила, фильтрация удалённых, NotFoundException |
| `PostRepository` | In-memory хранение, установка флага `removed` |
